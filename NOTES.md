# aotools

**For step-by-step install instructions and full command syntax written for end users, see `README.md`. This file is the build/development notes: what was tested, bugs found and fixed, and how to rebuild from source.**

A single-binary Go port of the ao486 DOS Toolkit (`mkvhd`, `resizevhd`, `mkmgl`, `mkchd`, `mkima`, `mountvhd`/`umountvhd`, `mountchd`/`umountchd`, plus the `make_mbr.py`/`fix_bpb.py`/`newgame_controller.py` helpers they drove). Ported from your real, running scripts on `/media/fat/linux` â€” nothing here was guessed from the README alone.

**Already deployed and smoke-tested on your MiSTer** at `/media/fat/linux/aotools/aotools` â€” none of the original files were touched; everything lives in that new folder. You can start using it right now over SSH.

**Everything is one file.** The `mountvhd`/`umountvhd`/`mountchd`/`umountchd`/`mkvhd`/`resizevhd`/`mkmgl`/`mkchd`/`mkima` shell functions used to live in a separate `aotools-functions.sh` you had to remember to source. They're now embedded directly inside the `aotools` binary itself and are wired up automatically by a one-time `aotools install` â€” there is no second file to install or track.

## One-time setup on your MiSTer

```
/media/fat/linux/aotools/aotools install
```

This appends a small marker-delimited block to `/media/fat/linux/user-startup.sh` (MiSTer's own persistent boot hook):

```
# --- aotools:begin (shell functions + PATH) ---
cat > /etc/profile.d/aotools.sh <<'AOTOOLS_PROFILE_EOF'
export PATH="$PATH:/media/fat/linux/aotools"
eval "$(/media/fat/linux/aotools/aotools shellinit)"
AOTOOLS_PROFILE_EOF
# --- aotools:end ---
```

Why it's shaped this way (this took a real bug to get right â€” see item 5 below): `user-startup.sh` runs once, as its own standalone child process at boot, so anything it `export`s or `eval`s directly dies with that one process and never reaches an SSH session opened afterward. What *does* reach every new login shell is `/etc/profile`, which on MiSTer already sources every `*.sh` file under `/etc/profile.d/`. So instead of exporting anything itself, the block above has `user-startup.sh` (re)write `/etc/profile.d/aotools.sh` â€” containing the real `export PATH=...` / `eval "$(... shellinit)"` lines â€” on every single boot. `install` also writes that same file immediately when you run it, so it works right away, without waiting for a reboot.

It's idempotent (checks for the begin marker and the current mechanism first, so running `install` again is a no-op) and only ever appends to `user-startup.sh` â€” it never rewrites or touches anything else already there. From the moment you run `install` onward, any *new* SSH session (no reboot required â€” though it also survives reboots) has `aotools` on `$PATH` (so `aotools <command>` works from any directory) and `mountvhd`/`umountvhd`/`mountchd`/`umountchd`/`mkvhd`/`resizevhd`/`mkmgl`/`mkchd`/`mkima` just available, no period, no sourcing, nothing else to remember.

If a machine already has an older install â€” either the very first format (shell functions only, no PATH) or the version right after that (which tried to export/eval directly in `user-startup.sh` and silently didn't work for new SSH sessions) â€” running `install` again detects that automatically, strips the old block, and replaces it with the current one above.

To get both in your *current* shell right now without opening a new session:

```
export PATH="$PATH:/media/fat/linux/aotools"
eval "$(/media/fat/linux/aotools/aotools shellinit)"
```

## Full command reference

```
aotools create vhd [-dos|-win31] <name.vhd> [archive|folder]
aotools resize vhd <name.vhd>
aotools create mgl -dos|-win31 "Display Name" [source_folder]
aotools create chd <input.iso|.cue|.bin|.gdi> <output.chd>
aotools create diskimage <name.ima> [source] [-s size]
aotools mount vhd <name.vhd>            (or `mountvhd <name.vhd>` via the shell function)
aotools umount vhd                      (or `umountvhd`)
aotools mount chd <name.chd>            (or `mountchd <name.chd>`)
aotools umount chd                      (or `umountchd`)
aotools mount diskimage <name.ima>
aotools umount diskimage
aotools install                         (one-time: wire shell functions + PATH into every future shell)
aotools uninstall                       (reverses install -- removes the user-startup.sh wiring only)
aotools shellinit                       (prints the shell functions; used internally by `install`)
aotools doctor                          (reports on qemu/chdman/templates/mtools/etc. -- see README.md)
```

Every command follows a consistent `<verb> <noun>` order: `create`/`resize`/`mount`/`umount` always come first, followed by what they act on (`vhd`/`mgl`/`chd`/`diskimage`). This was retrofitted from an earlier, inconsistent `<noun> <verb>` layout (`vhd create`, `mgl create`, `chd create`, `ima create`) â€” see the CLI syntax restructuring section below.

## Rebuilding it yourself

No compiler needed to *use* it â€” the binary at `/media/fat/linux/aotools/aotools` is a static ARMv7 executable, drop-in runnable. You'd only rebuild if you want to modify the source.

On the MiSTer itself (it has no compiler by default, but does have internet access):

```
curl -sSkL -o /tmp/go.tar.gz https://go.dev/dl/go1.26.5.linux-armv6l.tar.gz
mkdir -p /media/fat/linux/.mistercli_buildtools/goroot
tar -C /media/fat/linux/.mistercli_buildtools/goroot -xzf /tmp/go.tar.gz --strip-components=1
export GOROOT=/media/fat/linux/.mistercli_buildtools/goroot
export PATH=$GOROOT/bin:$PATH
export GOCACHE=/media/fat/linux/.mistercli_build/.gocache
export GOPATH=/media/fat/linux/.mistercli_build/.gopath
cd /media/fat/linux/.mistercli_build
go build -ldflags="-s -w" -o aotools .
```

(Directory names still say `mistercli` -- a leftover from before this tool was renamed to `aotools`; harmless, just a build-cache location, not user-facing. Also note: `GOROOT`/`PATH` must be set in separate `export` statements, not one combined `export A=... B=$A...` line -- bash expands `$GOROOT` against its value *before* that same command's assignments take effect, so a combined line silently uses the old value.)

First build compiles the whole standard library (a few minutes on ARMv7); rebuilds after that are seconds. **Extract the Go toolchain and set `GOCACHE`/`GOPATH` under `/media/fat`, not `/tmp`** â€” `/tmp` is a ~240MB tmpfs and is nowhere near big enough for a full Go toolchain + build cache.

Or cross-compile from any machine with Go installed:
```
GOOS=linux GOARCH=arm GOARM=7 CGO_ENABLED=0 go build -ldflags="-s -w" -o aotools .
```

## CLI syntax restructuring (verb-first order) and disk image mounting

The original command syntax put the file format before the action (`vhd create`, `vhd resize`, `mgl create`, `chd create`, `ima create`), while `mount`/`umount` were already action-first (`mount vhd`, `umount vhd`, etc.) -- an inconsistency the user flagged as confusing. All commands now consistently put the verb first: `create vhd`, `resize vhd`, `create mgl`, `create chd`, `create diskimage` (`mount`/`umount` were unchanged, since they were already in this order). `ima create` was also renamed to `create diskimage` throughout the CLI, matching the more descriptive naming already used for `vhd`/`mgl`/`chd`. Internal Go identifiers (`cmdIMACreate`, `imaMountPoint`, etc.) were deliberately left as-is; only the user-facing command name changed.

`create diskimage` also gained a size-selection prompt: it now always shows a numbered list of the five standard floppy sizes (360k/720k/1.2m/1.44m/2.88m), marks 1.44m as the default (selected by pressing Enter with no input) and, when a source is given, marks whichever size is the smallest one it fits as "recommended for source." An explicit `-s <size>` flag still skips the prompt entirely. Confirmed via `archext.go` that both zip archives and plain folders were already fully supported as `create diskimage` sources through the same `isDir`/`extractArchive` branching `create vhd`/`create mgl` use -- no new source-type handling was needed, only the prompt.

New commands `mount diskimage <name.ima>` / `umount diskimage` were added, matching `mount vhd`/`umount vhd` and `mount chd`/`umount chd` exactly: auto-`cd`'s into the mounted image on mount, auto-`cd`'s back to the starting directory on umount, and the `nano` wrapper's automatic `unix2dos` conversion now also covers files edited under the diskimage mount point (`/tmp/ima_mount`). Unlike VHDs, `.ima` files are raw "superfloppy" images with no partition table, so the mount itself is a plain `mount -o loop` with no partition offset, unlike the VHD case. No new legacy shortcut name (e.g. `mountima`) was added for this, since the original ao486 DOS Toolkit never had floppy-image mounting to begin with, and the broader direction of this change was to de-emphasize adding more shortcut names, not add to them.

`create vhd`'s `[archive]` source argument was also extended to accept a plain folder, matching `create diskimage` (user request, made mid-restructuring after noticing the inconsistency). Previously, `cmdVHDCreate`/`injectDOSArchive`/`createWin31VHD` (`vhdcreate.go`) only accepted `.zip`/`.tar`/`.tar.gz`/`.tar.bz2` via `isSupportedArchive`/`extractArchive`. A new `copyDirTree` helper (`util.go`) was added to recursively copy a plain folder's contents the same way `extractArchive` unpacks an archive's, and every `isSupportedArchive`/`extractArchive` call site in `vhdcreate.go` (source validation, size estimation, and both the DOS and Windows 3.1 injection paths) now branches on `isDir(source)` first. The "delete the original source?" prompt after injection is skipped entirely for folder sources -- offering to `rm -rf` a folder the user pointed at (which could be their live working directory) is a materially different risk than offering to delete a redundant `.zip`, so it's not offered at all rather than defaulting to "no." Verified end-to-end on the real device: a `create vhd -dos` run against a small test folder (containing a `RUN.BAT`) completed the full QEMU-driven DOS install, correctly copied the folder's contents into `\<GAMENAME>` via `copyDirTree` (confirmed by mounting the resulting VHD and inspecting the file), updated `AUTOEXEC.BAT` with `CD <GAMENAME>`, and left the original source folder completely untouched afterward.

## Nine real bugs found and fixed during testing (worth knowing about)

1. **`vhd resize`**: writing `IO.SYS`/`MSDOS.SYS`/`COMMAND.COM` via a single multi-file `mtools` batch (matching the original bash script's own approach) came back with a corrupted `COMMAND.COM` on one real run â€” partly zeroed, wrong FAT chain â€” even though the identical batch succeeded in isolated retries. Fixed by routing those three files through the same reliable kernel loop-mount + `cp` path used everywhere else in the codebase, instead of raw `mtools`.
2. **Archive injection**: the wrapper-folder-flattening step failed with "invalid cross-device link" because its staging directory defaulted to `/tmp` while the archive staging area lives on `/media/fat` â€” a different filesystem, which `os.Rename` (unlike the `mv` *command*, which silently falls back to copy+delete) can't cross. Fixed by staging as a sibling directory instead.
3. **`aotools install` idempotency**: the first version checked for a literal `"aotools shellinit"` substring in `user-startup.sh` to decide whether it was already installed, assuming the binary would always be invoked as `aotools`. Testing with a differently-named test build (`aotools_new`) showed the check silently failed and re-appended a duplicate block, since the actual written line was `.../aotools_new shellinit`, not `.../aotools shellinit`. Fixed by checking for the fixed comment-header line instead, which doesn't depend on the binary's own filename. Re-verified: running `install` twice now only ever appends once.
4. **`promptLine` losing input across sequential prompts**: `promptLine` (used everywhere aotools asks the user a question) created a brand-new `bufio.Reader` on stdin every single call. That's invisible with a real interactive terminal (input arrives one line at a time), but breaks with piped/scripted input spanning multiple lines: the first call's `bufio.Reader` buffers ahead and silently swallows everything after the first line, then gets thrown away, so every prompt after the first one comes back empty. Caught directly while testing the new install-time download flow (which asks two questions in a row -- "download?" then the "I agree" acknowledgment) with piped test input: the second answer kept coming back as if the user had pressed Enter with nothing typed, incorrectly cancelling the download every time. Fixed by sharing one `bufio.Reader` across every `promptLine` call instead of creating a new one each time. Re-verified with the same two-prompt flow: both answers are now read correctly.
5. **`install`'s PATH export never actually reached new SSH sessions**: the first version of the PATH feature had `install` write `export PATH=...` / `eval "$(... shellinit)"` directly into `user-startup.sh`. This looked correct when tested via a `bash -c` subshell spawned from within the same already-live SSH session (it inherited the parent shell's already-exported PATH), but that test was flawed -- it didn't prove anything about a genuinely new login. On the real device, after a real reboot and a real new SSH session, `aotools <command>` still reported "command not found." Root cause: `user-startup.sh` is run once, as a standalone child process, by `/etc/init.d/S99user` at boot; anything it exports dies with that one process and is never inherited by any shell started afterward, reboot or not. Fixed by having `install` stop exporting/evaling inside `user-startup.sh` entirely -- instead, `user-startup.sh` now (re)writes `/etc/profile.d/aotools.sh` on every boot (containing the real export/eval lines), since `/etc/profile` -- which MiSTer's login shells genuinely do source -- loops over and sources every `*.sh` file in `/etc/profile.d/`. `install` also writes that file immediately for instant effect, no reboot needed. Re-verified via `bash -lc '...'` (a genuine login shell, matching a real interactive SSH terminal) that PATH and the shell functions are both present in a fresh session, and that the user's exact originally-failing command now succeeds.
6. **`aotools mount vhd`/`aotools umount vhd` (typed directly) never actually `cd`'d anywhere, unlike `mountvhd`/`umountvhd`**: flagged by the user as unacceptable -- the whole point of a unified `aotools` command was for it to behave the same way everywhere, and having `aotools mount vhd x.vhd` silently *not* `cd` you in (while `mountvhd x.vhd` did) was two different behaviors under one name. Worse, mixing the two forms in one session corrupted the shared "previous directory" state: mounting via the raw binary form then unmounting via the shell function landed you in `/tmp` instead of back where you started, because the raw path never set the `AOTOOLS_VHD_PREV_DIR`/`AOTOOLS_CHD_PREV_DIR` variable the shell function reads on umount. Root cause: `aotools mount vhd`, typed at the prompt, ran the compiled binary directly -- a subprocess, which can never change its parent shell's working directory; only `mountvhd`'s shell-function wrapper could actually `cd`. Fixed by making `aotools` itself a shell function (in `shellFunctionsTemplate`, `shellinit.go`): it special-cases `mount vhd`/`umount vhd`/`mount chd`/`umount chd` to do the `cd` and track the previous directory, and passes every other subcommand straight through to the real binary unmodified. `mountvhd`/`umountvhd`/`mountchd`/`umountchd`/`mkvhd`/etc. are now thin wrappers that call this same `aotools` function, so every entry point shares one code path and one set of state variables -- there is no way left to end up in an inconsistent state by mixing spellings. Verified on the real device (using the smaller `ultima8.chd` for speed on the CHD side) with every combination: `aotools mount/umount vhd`, legacy `mountvhd`/`umountvhd`, mixed mount-via-one-form/umount-via-the-other, and the same four combinations again for `chd` -- every case correctly `cd`'d in on mount and back to the original starting directory on umount.
5. **`resize vhd`: root directory entry order not preserved, breaking boot on real DOS/BIOS hardware**: found during a comprehensive real-hardware test pass across every file-management command. A resized VHD passed the tool's own internal byte-for-byte file-content verification but failed to boot with "Non-System disk or disk error" -- reproduced twice, including once under a completely clean, fully isolated retry with zero possibility of interference from other processes, which ruled out a testing-methodology explanation. Root cause: MS-DOS 6.22's boot sector code loads `IO.SYS`/`MSDOS.SYS` by root-directory-entry POSITION, not by name search -- `IO.SYS` must be the very first root directory entry (`MSDOS.SYS` second) or the disk won't boot, even though every file's content and the boot sector's own code/BPB are perfectly valid. `cmdVHDResize` (`vhdresize.go`) already special-cased copying `IO.SYS`/`MSDOS.SYS`/`COMMAND.COM` first for exactly this reason, but built that list by iterating `os.ReadDir`'s alphabetically-sorted `entries` and checking membership, which silently produced the list in alphabetical order (`COMMAND.COM`, `IO.SYS`, `MSDOS.SYS`) instead of canonical DOS boot order -- confirmed directly via a raw kernel loop-mount directory listing showing `COMMAND.COM` as entry 1. Fixed by building the system-files list by iterating the fixed `sysFiles` order (`IO.SYS`, `MSDOS.SYS`, `COMMAND.COM`) and searching `entries` for each by name, instead of the reverse. Re-verified on real hardware across three resize sizes (40MB, 100MB, 2047MB) plus a fresh directory-listing check confirming `IO.SYS` lands as entry 1 every time.
6. **`resize vhd`: boot sector merge and file content silently lost on large (near-2047MB) resizes**: found immediately after fixing bug 5 above, while boot-testing the largest resize size specifically. The BIOS reported "This is not a bootable disk" (a different, lower-level failure than bug 5's), and inspection showed the final boot sector still had `mkfs.vfat`'s own untouched defaults (OEM string `mkfs.fat`, 255 heads) instead of the merged real-DOS values -- even though both `fixBPB` and `writeBootSectorMerge` had logged success. Moving the boot sector write to happen after all `loopCopyIn` calls (rather than before, as originally written) surfaced a second, more serious symptom on retest: `IO.SYS`/`MSDOS.SYS`/`COMMAND.COM` were completely missing from the final file -- confirmed via a raw kernel loop-mount listing, not just an mtools quirk. The original code made two separate `loopCopyIn` calls (one for the three system files, one for everything else), each its own independent loop-attach/mount/copy/unmount/detach cycle; the pattern of "first call's writes lost, second call's writes intact" only makes sense if the second call's fresh loop-attach got a stale, pre-first-call view of the root directory on this specific embedded kernel/loop-driver combination, and overwrote the first call's directory entries as if their slots were still free. Fixed by merging both copies into a single `loopCopyIn` call (one combined, order-preserving `sysItems` + `remainingItems` list, one mount session) and moving the boot sector merge to run last, after that single copy call and before the mtools-only (non-loop) attribute-application and verification steps -- removing both the second loop-attach cycle and any loop-device activity after the direct boot-sector write. Re-verified on real hardware at 2047MB: directory listing shows all three system files present in correct order, the boot sector shows the correct merged `MSDOS5.0` OEM string and 16 heads, `resize vhd`'s own verification now genuinely passes (previously it was passing on a false basis, before the file-loss symptom was even visible), and the VHD boots to the game splash screen.
7. **`create vhd -win31`: the identical bugs 5 and 6, in `createWin31VHD`**: found by inspection immediately after fixing bugs 5/6, on the hunch that `createWin31VHD` (`vhdcreate.go`) clones its template through the exact same format-then-copy-then-merge-boot-sector pattern as `resize vhd`, rather than via a real QEMU-driven install (which is why fresh `-dos` VHDs were never affected by bugs 5/6 in the first place -- QEMU's own FORMAT/SYS writes real system files directly). Confirmed on reading: `createWin31VHD` built its system-files list by iterating alphabetically-sorted `entries` (same bug as 5), used two separate `loopCopyIn` calls for the main content plus a third for archive/game injection (same class of bug as 6), and ran `writeBootSectorMerge` before all of them. Fixed with the identical pattern used for `resize vhd`: system files list now built by iterating the fixed `sysFiles` order; the main content copy (system files + everything else) now goes through one combined `loopCopyIn` call; and the boot sector merge now runs dead last, after the main content copy AND the archive/game injection copy (if any), with nothing loop-based touching the file afterward. Verified end-to-end on real hardware: cloned a real Windows 3.1 game (Civilization's `CIVWIN` folder, copied out of the user's own existing, untouched `civilization w3-1.vhd` for genuine test content) via `create vhd -win31` with a folder source, confirmed correct `IO.SYS`/`MSDOS.SYS`/`COMMAND.COM` order and a correctly-merged boot sector (`MSDOS5.0` OEM, matching BPB) via direct inspection, and confirmed DOS boots correctly before Windows 3.1 itself takes over (visual confirmation of the Windows GUI stage itself was not reliable through the available screenshot tooling -- see the note under bug 8).
8. **`resize vhd` (and `create diskimage`'s archive-staging path): staging directories defaulted to `/tmp`, a small tmpfs, instead of the large `/media/fat` scratch area**: found while resizing a Windows 3.1 VHD (more files and larger total content than the plain-DOS VHDs bugs 5-7 were tested against). The resize failed with "No space left on device" during its own verification step. `/tmp` on this MiSTer is a ~240MB tmpfs (see the Known limitation note below for why it fills up), and `cmdVHDResize`'s `stageDir`/`verifyDir` (`vhdresize.go`) and `cmdIMACreate`'s archive-staging directory (`ima.go`) all used a plain `os.MkdirTemp("", ...)`, which defaults to `/tmp`, instead of the project's own `mktempBig()` helper (already used correctly by `createWin31VHD`/`injectDOSArchive` for this exact reason -- staging real file content needs real disk space, not RAM-backed tmpfs). Fixed by switching all three spots to `mktempBig()`, which stages under a scratch directory on `/media/fat` instead. `loopfs.go`'s `loopCopyIn` mount-point directory and `vhdcreate.go`'s `dosOverheadBytes` mount-point directory were deliberately left alone -- those are just empty mount points for a loop-mounted filesystem, not bulk-copy targets, so they carry none of this risk. Re-verified: the same Windows 3.1 resize that previously failed now completes and verifies successfully, and `/tmp` usage stayed under 10MB throughout instead of climbing into the hundreds of megabytes.

9. **`fatal()`'s `os.Exit(1)` skipped deferred cleanup, leaking scratch staging directories (and, on `resize vhd`, the half-built replacement container) on any failure partway through**: this was the documented "Known limitation" below, promoted to a real fix once directly observed to matter -- see the previous entry (bug 8), where `/tmp` filled up specifically because of orphaned `stageDir`/`verifyDir` directories left behind by earlier failed test runs in the same session. Root cause: Go's `os.Exit()` terminates the process immediately and skips every registered `defer` on the call stack, so any function relying on `defer os.RemoveAll(stageDir)` (or, in `resize vhd`'s case, `defer os.Remove(newVHD)` for the half-built replacement container) never got to run whenever a downstream `fatal()` fired. Fixed by changing `fatal()` (`util.go`) to `panic(fatalExit{})` instead of calling `os.Exit(1)` directly -- a `panic` unwinds the call stack and runs every `defer` along the way before anything else happens, which is exactly the cleanup guarantee `os.Exit` doesn't provide. `main()` (`main.go`) was restructured so the original dispatch logic now lives in a `run()` function, called from a new `main()` that defers a `recover()`: if the recovered value is a `fatalExit`, it exits with status 1 (matching the original behavior exactly, after cleanup has already run); any other panic value is re-panicked immediately, so a genuine bug still crashes visibly with a real stack trace instead of being mistaken for a clean "Error: ..." exit. Re-verified on real hardware by forcing a `fatal()` call after staging directories were already created and populated, for all three affected commands: `resize vhd` (non-numeric size input), `create vhd -win31` (same), and `create diskimage` (a corrupt archive source) -- in every case the same error message and exit code 1 as before, but with the scratch directory actually gone afterward instead of left behind. Also spot-checked with a standalone test program that a genuine, unrelated panic (an out-of-range index, simulating a real bug) is still re-panicked and crashes with its full stack trace and a non-1 exit code, confirming the fix doesn't mask real bugs as clean exits.

## Performance fix

`mount chd`'s raw-sector stripping step (`stripRawSectors`, used for MODE1/2352 and MODE2/2352 CD images) originally read/wrote one 2352-byte sector at a time with unbuffered `os.File` calls â€” roughly 350,000 syscall pairs for a typical CD image, which measured out to only ~1MB/s on real ARM hardware. Wrapped both sides in 1MB `bufio` buffers, which brought it up to tens of MB/s.

## Formerly a known limitation (now fixed -- see bug 9 above)

Error paths that call `fatal()` used to skip Go's deferred cleanup (since `fatal()` called `os.Exit` directly), so a failed `vhd resize` or archive injection could leave a scratch temp file/directory behind instead of cleaning up immediately (harmless clutter under `/tmp` or `/media/fat/linux/.mkvhd_scratch`, not data loss -- nothing user-facing was ever corrupted). This is now fixed via the `panic`/`recover` mechanism described in bug 9 above; failures partway through `resize vhd`, `create vhd -win31`, and `create diskimage` now clean up their staging directories immediately instead of leaving them behind.

## Note on screenshot-based boot verification and Windows 3.1 / graphical modes

During this testing pass, the MiSTer's native screenshot mechanism (used to visually confirm a game reached its title screen after booting) produced a consistent, garbled, striped pattern for every Windows 3.1 boot -- both for newly-created test VHDs and for the user's own real, pre-existing, already-working Windows 3.1 game library. The pattern was reproducible, changed in response to real keyboard input (confirming the core itself was alive and rendering, not hung), and is a known limitation of that capture path for this particular video mode/resolution -- not a defect in any `aotools`-created VHD. Boot correctness for these cases was instead confirmed at the disk level (root directory order, boot sector bytes, MBR/BPB fields) plus confirmation that plain MS-DOS boots cleanly before Windows 3.1 itself takes over. An OBS-based screenshot tool (which captures the actual video output rather than decoding the core's internal framebuffer) is available as a more reliable alternative for this specific class of verification in the future.
